# BUAA CO 课程仓库

该仓库仅供参考。2026年这个仓库的主人选修了“处理器设计实训课程”，发现之前在CO课程提交的流水线CPU存在诸多问题，最突出的应当是转发机制设计不合理。即便如此，这些代码仍然通过了P5/P6的强测；P7中断强测有一个数据点未通过，目前不知产生原因。

## 作业代码部分（`cpu_code` 目录）

这些代码中可能会展现编写者恶劣的代码习惯，也有一些架构不够合理 / 冗余，请大家批判性地阅读。还是建议将流水寄存器以外的各个器件进行整合，命名为`{流水级}_stage`格式，不要像仓库里的代码一样，让各个部件的实例化代码散落在不同位置。

无论如何也不要将这些代码直接复制变成你的作业！CO 课程的查重会覆盖**历届**提交内容，抄袭我的代码会导致作业 0 分，也不利于你理解 CPU 架构，从而更不利于完成课上新增指令的考核。（助教也许还会嫌弃你抄代码都不知道抄个靠谱一点的）

## 评测机部分（`cpu_check_px` 目录）

### 说明

这些评测机能够生成测试数据以评测自行搭建的 CPU。不过本人不能保证数据具有足够的强度，所以也请自行构造测试数据来检验。

P3 的评测机只能对拍，P4 及之后的评测机可实现“自测”。有兴趣的同学可以尝试解析 Logisim 输出的二进制文件来实现 P3 的“自测”。

由于CO课程时间紧任务重，评测机的代码质量比较低，只能保证完成基础的功能。同届还有同学构造了更合理的评测机，你们在讨论区的“往届精华帖”中也许就能看到。

### 鸣谢

感谢 Tony 学长等人开发的魔改版 Mars！它大大减少了我们实现自测的工作量。

### 局限性

1. P3 的评测机数据较弱，且并不总是合法，容易出现死循环的状况。建议用 P4 及以后评测机目录下的 `generator.py` 文件来替换 P3 目录下的同名文件。

2. 由于 MARS 本身的局限性，P7 仅能实现对异常的检查，中断部分的测试请与同学对拍。此外，在 MARS 中写地址 0x2fff 会被判定为异常，但在课程作业中这并不是异常，请辩证看待 MARS 运行结果。
